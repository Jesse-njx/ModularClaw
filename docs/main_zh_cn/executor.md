# Executor 模块（简明指南）

本页用通俗语言解释 **`modules/executor.py`**：它做什么，以及**为什么 Sender 会等它完成**。

你**不需要**先理解整个框架，也能看懂这里的内容。

---

## Executor 是做什么的？

把 AI 想象成一个人，他会写一张**特定格式的小便条**。便条上的意思是：“请执行这条 shell 命令。”

**Executor** 是程序里负责这些事情的部分：

1. 在对话内容中找到这些便条。
2. 在计算机上执行命令。
3. 把执行结果写回对话，供 AI 后续读取。

如果 Executor 正忙，或者还在等待命令执行结束，**就不应该再次调用 AI**。这是 Executor 对 Sender 的主要“阻塞”方式，而且这是有益的阻塞。

---

## 这种便条长什么样？

Executor 只关心满足以下条件的上下文项：

- 来自 AI 回复的普通文本，**并且**
- 标记为 JSON（`label` 为 `json`），**并且**
- 解析后 JSON 是一个**字典**，且 `type` = `"tool_call"`、`name` = `"execute_command"`。

简化示例：

```json
{
  "type": "tool_call",
  "name": "execute_command",
  "arguments": { "command": "ls" }
}
```

所以 AI 的回答不只是聊天文本，也可能包含这类 JSON。Executor 会在每次 tick 中检查它们。

---

## 分步骤看：发生了什么？

1. **识别工具调用**  
   Executor 扫描 session 的 context 列表。发现上述 JSON 且该位置尚未被“锁定”时，就会接管它。

2. **声明占用该位置**  
   它会把这个索引标记为**已占用（claimed）**，避免同一条命令被重复执行。

3. **后台执行命令**  
   命令在**独立线程**中运行，避免程序卡住。执行期间，Executor 认为工作**尚未完成**。

4. **把结果写回去**  
   命令结束后，Executor 会把原 JSON 块**替换**为一个**工具结果**（context 中的另一种类型）。它还会解除占用，并设置一个标记表示：**需要再跑一轮循环**（让系统继续推进）。

5. **告知 Sender “等待”或“可以”**  
   Executor 会把自己的 `"Ready to send"` 状态设置为 **`pending`**（还有事要做）或 **`ready`**（没有待执行项）。  
   **Sender** 会等待，直到**所有**其他模块对同一个状态键都返回 **`ready`**。

所以：**只要命令仍在执行，或仍有工具调用待处理，Executor 就会报 `pending`，Sender 就不会发送。**

---

## 它与 Sender 如何关联（简版）

- **Sender** 会问：“大家都准备好发送了吗？”  
- **Executor** 在还有命令待执行或正在执行时会回答 **`pending`**。  
- 全部处理完后，Executor 才会回答 **`ready`**。  
- 只有这样 Sender 才能继续（同时还要通过其他模块的同类检查）。

更多细节可见 [`sender.md`](sender.md)。

---

## 补充：工具提示词

会话开始时，Executor 可以把配置中的一段文本（executor 配置里的 `prompt`）加入 context。这段文本用于教模型**如何**格式化工具调用。你可以把它理解为“启动时贴进聊天里的说明”。

---

## 安全提示

Executor 会对模型给出的字符串使用 **`shell=True`** 执行命令。这在真实部署中能力很强，也有风险。本文只解释它**如何**与 Sender 交互；安全加固是另一个独立话题。
